Git 教程学习笔记

发表于 2018-07-21 | 更新于 2018-07-25 | 分类于 代码管理
来源：Git 教程 - 廖雪峰的官方网站
  • 2005 年，Linus 花了两周时间用 C 写了一个分布式版本控制系统，Git。2008 年，GitHub 上线。GitHub 为开源项目免费提供 Git 存储。2018 年，GitHub 被微软收购。
  • 使用 Git 的理由：方便管理代码的修改，通过不同版本之间的对比，可以明确的知道何处发生改动（增、删、改）。
  • 集中式版本控制系统与分布式版本控制系统的对比：集中式vs分布式
    • 集中式：版本库集中放在服务器，本地没有历史版本，每次 commit 都必须联网，否则无法记录修改。使用体验受网速影响。如果服务器挂了，所有人停止工作。
    • 分布式：每一台电脑上都有一个完整的版本库，安全性比集中式高。实际工作中，分布式也有一台充当“中央服务器”的电脑，方便不同电脑之间交换修改。可以在断网的条件下，正常进行开发工作，本地版本库的版本提交不受影响，支持版本回退，只是无法推送到远程仓库（“中央服务器”）。Git 只在 push/pull 操作时需要联网。在实际工作中，commit 操作频率相对更高。“中央服务器”如果挂了，所有人仍正常工作，仅仅是不方便交换修改而已。可以随时创建一个新的中央库，从任一台电脑同步之后就可以恢复中央库了。
    • CVS：最早的开源、免费的集中式版本控制系统，存在提交文件不完整，版本库会突然出现损坏的情况。
    • SVN：目前用的最多的开源、免费的集中式版本控制系统，比 CVS 稳定。
    • Git：目前最流行的分布式版本控制系统。分为：
      • 工作区（Working Directory）：执行 git init 初始化时的本地目录。
      • 版本库（又名仓库，Repository）：执行 git init 初始化时，会在当前本地目录下生成一个文件夹 .git，默认隐藏。.git 就是 Git 的本地版本库。
        • 暂存区（stage 或叫 index）：git add <file> 即是将工作区的文件变动添加到暂存区。
        • master 分支：由 Git 自动创建：git commit -m <message> 即是将暂存区的文件变动提交到当前分支。
        • HEAD 指针： 指向当前分支。
      • 示意图：工作区、版本库（暂存区、master 分支、HEAD 指针）

Git和其他版本控制系统如SVN的一个不同之处就是有暂存区的概念。

  • git config
    显示与配置相关的命令参数。

  • git config -l
    显示所有配置。

  • git init
    初始化一个目录，通知 Git 对目录进行版本追踪管理。

  • git add <file>
    从工作区添加到暂存区。示意图

  • git commit -m <message>
    从暂存区提交到本地版本库。示意图

  • git status
    查看工作区状态，显示修改了哪些文件。

  • git status -sb
    作用与 git status 相同，只是显示信息相对更加简化，方便查看。

  • git diff [<file>]
    查看文件具体的修改内容。工作区(working directory)和暂存区(stage)的比较。

  • git diff --cached
    查看暂存区(stage)和分支(master)的比较。

  • git diff HEAD -- <file>
    查看工作区和版本库里面最新版本的区别。

  • git log
    由近及远查看提交记录。(commit 时的 message)

  • git reflog
    由近及远查看命令记录。(使用过的 Git 操作命令)

  • git log --pretty=oneline
    作用与 git log 相同，但是 --pretty=oneline 参数会简化显示信息。
    加上 --abbrev-commit 参数会使 commit_id 较变短，只有 7 位。

  • git log --graph
    查看分支合并图。

  • git log --graph --pretty=oneline --abbrev-commit
    作用同 git log --graph，但是信息会更简化。

  • git reset --hard <commit_id>
    回退到指定版本。commit_id 也可以是 HEAD 关键字。HEAD 指代当前版本，HEAD^ 指代上一个版本，HEAD^^ 指代上上个版本，以此类推。git reset --hard HEAD^100 指回退到前第 100 个版本。

  • git reset HEAD <file>
    撤销暂存区的修改（unstage），重新放回工作区。
    git reset 命令既可以回退版本，也可以把暂存区的修改回退到工作区。

  • git checkout -- <file>
    撤销工作区的修改。
    本质：用版本库里的版本替换工作区的版本。
    • 工作区修改未提交到暂存区，则会使工作区与版本库的最新版本保持一致。
    • 工作区修改已经提交到暂存区，则会使工作区与暂存区的版本保持一致。
  • git checkout <branch>
    切换到另一个分支。注意没有 -- 参数。
  • 删除文件
    • rm <file>
      先删除本地文件。
    • git rm <file>
      再删除版本库里的文件。这一步类似 git add <file> 操作。
    • git commit -m <message>
      记得提交，提交后才能真正删除文件。
  • 误删文件
    • git checkout -- <file> 用版本库里的版本替换工作区的版本。因此会丢失最近一次提交后修改的内容。
  • 关联本地 Git 仓库和远程仓库
    • 本地 Git 仓库和 GitHub 仓库之间的传输是通过 SSH 加密的。
    • 创建SSH Key
    • 添加远程库
      • 远程库的名字: origin
      • git remote add origin git@server-name:path/repo-name.git
      • 命令格式：git remote add 远程库的名字 远程库的地址
        将本地仓库关联到一个远程库。
      • git push -u origin master
      • 命令格式：git push -u 远程库的名字 要推送到的远程库中的哪个分支
        第一次推送时加上 -u 参数。
        1. 将本地 master 分支内容推送到远程库的 master 分支。
        2. 关联本地 master 分支和远程库 master 分支。在以后的推送或者拉取时就可以简化命令。
      • git push
        把当前分支推送到远程库。
  • 从远程库克隆
    • git clone git@server-name:path/repo-name.git
    • cd "repo-name"
    • Git 支持多种协议，包括 https，但通过 ssh 支持的原生 git 协议速度最快。
  • 创建与合并分支
    • HEAD 指针： 指向当前分支。
    • git checkout -b <branch>
      创建并切换到新的分支。<branch> 指代分支的名字。
      相当于以下两条命令
      • git branch <branch> 创建新的分支。
      • git checkout <branch> 切换到另一个分支。
    • git branch
      列出所有分支。
      前面标有 * 的是即是当前分支。
    • git merge <branch>
      合并指定分支 <branch> 到当前分支。
      故执行合并 git merge 前，需要先 git checkout 切换到当前分支。
    • git branch -d <branch>
      删除分支。
    • Git 鼓励使用分支完成某个任务，合并后再删掉分支，这和直接在 master 分支上工作效果是一样的，但过程更安全。
  • 解决冲突
  • 分支管理策略
    • 团队合作示意图
    • 合并分支时，默认用 Fast forward 模式。该模式下，删除分支后，会丢掉分支信息。
    • 合并分支时，加上 --no-ff 参数，强制禁用 Fast forward 模式，改用普通模式合并。Git 会在 merge 时生成一个新的 commit，这样，从分支历史上就可以看出分支信息，能看出来曾经做过合并。而 fast forward 合并就看不出来曾经做过合并。
    • git merge --no-ff -m <message> <branch>
      使用普通模式合并，并提交一个新的 commit。
  • Bug 分支
    • git stash
      把当前工作现场“储藏”起来。
      此时工作区就被清空了。（working tree clean）
    • git stash list
      查看被暂存的工作现场。
    • 恢复暂存起来的工作现场到工作区
      • git stash apply 恢复的同时 stash 内容并不删除。需要用 git stash drop 来删除。
      • git stash pop 恢复的同时把 stash 内容也删了。
      • git stash apply stash@{0} 恢复指定的 stash 到工作区。
  • Feature分支
    • git branch -D <branch> 强制删除未合并的分支。
    • git branch -d <brabch> 删除已合并的分支。
  • 多人协作
    • git remote
      查看远程库信息。
      加上 -v 参数，可以查看更详细的信息（push、fetch 各自的地址）。
    • git push origin <branch>
      推送指定的分支到远程库。
    • git checkout -b <branch> origin/<branch>
      在本地创建和远程分支对应的分支。
    • git branch --set-upstream-to=origin/<branch> <branch>
      建立本地分支和远程分支的关联。
    • git pull
      从远程抓取分支。
  • 标签管理

    标签是版本库的一个快照。
    标签就是一个让人容易记住的有意义的名字，它跟某个 commit 绑在一起。
    分支可以移动，标签不能移动。

  • 创建标签
    • git tag <tagname> 创建标签。默认为 HEAD，也可以指定一个 commit_id。
    • git tag 查看所有标签。
    • git tag -a <tagname> -m <message>
      指定标签信息。默认为 HEAD，也可以指定一个 commit_id。
    • git show <tagname> 查看标签信息。
